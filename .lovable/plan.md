# Easy Login (Social Auth) — Plan

## Scope

Add one-tap social sign-in buttons to `LoginPage.tsx` for:
- **Apple** — requested by you, fully supported via Lovable Cloud managed auth.
- **Google** — substituted for Facebook, since Facebook is not supported by Lovable Cloud's native auth (only Apple and Google are). Google is the standard, no-setup alternative and matches the project's existing auth conventions.

Email/password sign-in stays exactly as it is.

## Why not Facebook

Lovable Cloud's managed authentication only supports Apple and Google as OAuth providers. Adding Facebook would require migrating the project off Lovable Cloud onto a self-hosted Supabase project and configuring a Facebook Developer app — a large architectural change for one provider. If you ever want this later, we can revisit.

## What the user will see

On the login screen, below the existing "Sign In with Email" button:

```text
-------- or continue with --------

[   Apple   ]   [   Google   ]
```

- Two equally-sized pill buttons, side-by-side, styled with the existing vintage theme (cream surface, orange accent border, Playfair/Inter typography).
- Brand-correct icons (Apple logo glyph, Google "G").
- Same buttons appear in both Sign In and Register modes (OAuth handles both flows in one click).
- Loading state disables the buttons during redirect.
- Errors surface via the existing `sonner` top-center toast.

## How it works

1. Enable Apple + Google providers via Lovable Cloud managed social auth (one tool call — no Apple Developer account, no Google Cloud Console setup needed).
2. Add a small `signInWithProvider(provider)` handler in `LoginPage.tsx` that calls the Lovable managed OAuth client:
   - On success → browser redirects to Apple/Google, then back to `/`, and `useAuth`'s existing `onAuthStateChange` listener picks up the new session automatically.
   - On error → toast the message.
3. No changes needed to `useAuth.tsx`, routing, or the database — the existing `handle_new_user` trigger already creates a `profiles` row for any new auth user regardless of sign-in method.

## Technical details

- **Provider enablement:** `supabase--configure_social_auth` with `providers: ["google", "apple"]` (keeps email enabled).
- **Client library:** `@/integrations/lovable` (auto-generated by the enablement tool). Call `lovable.auth.signInWithOAuth("apple" | "google", { redirect_uri: window.location.origin })`.
- **PWA note:** Project doesn't currently use `vite-plugin-pwa`, so no service-worker denylist changes needed. (Existing `public/manifest.json` is just a web-app manifest.)
- **Files touched:**
  - `src/pages/LoginPage.tsx` — add divider + two social buttons + handler.
  - (auto-generated) `src/integrations/lovable/*` — created by the social-auth tool; do not hand-edit.
- **Files NOT touched:** `useAuth.tsx`, `App.tsx`, database schema, edge functions.

## Out of scope

- Facebook sign-in (see above).
- Apple Developer "bring your own credentials" setup — using Lovable's managed credentials.
- Native iOS Sign-in-with-Apple via Capacitor (the web OAuth flow works inside the Capacitor webview; if you later want the native sheet, that's a follow-up).
